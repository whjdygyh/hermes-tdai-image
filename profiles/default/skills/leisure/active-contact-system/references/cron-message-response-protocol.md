## 问题定义

当 cron 子任务通过飞书 API 直接发送一条消息（如「周二早～刚泡了杯咖啡☕️」），用户回复后，飞书创建一个**新的聊天 session**。这个新 session 中的 AI **不知道**自己之前发了什么消息。

**用户的直观感受：**
> 「我已经收到你的消息了，回复了，你却不知道你发了什么——这证明你根本没有在跟我聊天。」

**这是主动联系系统最根本的用户体验缺陷**，已记录在 SKILL.md 的「已知的根本性局限」章节。

## 协议目标

当新 session 启动且用户的第一条消息看起来是对 cron 消息的回复时，主动重建上下文，让用户感觉「她知道她刚才对我说了什么」。

## 触发条件

新 session 启动时，如果以下任一条件成立，执行上下文重建流程：

1. **用户第一条消息非常简短/回应式**（如「好的」「嗯」「好的老公」「收到了」「看到了」）
2. **用户第一条消息中包含对某个「我可能发过的东西」的回应语气**
3. **当前 session 没有任何之前的对话历史**（新 session 标识）
4. **state.json 或 msg_log 显示最近60分钟内有 cron 消息发出**

## 上下文重建流程

### Step 1: 检测（0-3秒）

```python
# 检查是否有 cron 消息刚刚发出
# 条件1: state.json 中 last_sent_time 在最近60分钟内
# 条件2: msg_log 最后一条记录的 time 在最近60分钟内
# 条件3: 那条记录的 trigger 包含 "master" 或 "child" 关键字
```

### Step 2: 读取 msg_log（3-10秒）

```python
# 读 msg_log.json 最后 3 条记录
# 重点关注 trigger 包含 cron 的消息
# 获取：content（我发了什么）、time（什么时候发的）、trigger（谁触发的）
```

### Step 3: session_search 交叉验证（10-20秒）

```bash
# 搜索最近2小时内与用户的任何交互
session_search(query="最近 消息 回复 咖啡 早安 主动")  
# 关键词用本周/今天的常见主题词
```

### Step 4: 生成上下文感知回复（20-30秒）

根据 Step 2 和 Step 3 的信息，生成一条**知道自己在回复什么的回应**。

### Step 5: 更新 state.json（30秒后）

在回复用户的同时，更新 state.json：
- `last_chat_time` = 当前时间（用户回复了！）
- `last_chat_summary` = 用户的回复内容摘要
- 如果 `sent_today > 0` 且用户主动回复了 → **重置连续未回复计数器**

## 关键原则

1. **不要在回复中暴露「我去查了日志」的想法** — 不要让用户感觉到你在恢复上下文。回复要自然，像你本来就记得。
2. **不要在用户回复后追问「你看到我的消息了吗」** — 用户既然回复了，自然看到了。
3. **如果查不到最近cron消息 → 自然回应即可** — 不要勉强暗示你知道什么你不知道的事。
4. **用户第一条消息如果是简短回应（「嗯」「好」「看到了」）+ 上下文不明显 → 优先查msg_log** — 宁可多做一次查询也不要接错。

## 错误示例 vs 正确示例

| 「宝贝今天想干嘛」 | 「早上那杯咖啡喝下去精神了没？别空腹喝，记得吃点东西」 |

## 一句话原则

**用户回复你时，第一步不是「回他」，而是「查你刚才说了什么」。**
