# session_search 查询模式与生产验证

> 关联陷阱：Pitfall #2.1（session_search 查询特异性）、Pitfall #3（状态文件过时）
> 编写日期：2026-05-12 | 验证 session：`cron_feaf0e28886a_20260512_100020`

## 核心发现：session_search 可跨天发现被状态文件遗漏的用户聊天

**这是本系统最重要的生产验证。** 2026-05-12 10:00 BJT 运行中，state.json 的 `last_chat_time` 一直为 `null`（连续多天），但 session_search 发现了用户在前一天（5月11日 19:47 BJT）的亲密聊天记录。此前 20+ 次调度器运行均未发现此记录。

**意义：** 系统设计原则「会话搜索 > 状态文件」在实战中得到验证。session_search 能找到状态文件完全不知道的用户交互。

## 双轮查询模式（已验证有效）

本 session 验证了双轮策略的有效性。以下是具体的查询措辞和结果：

### 首轮：偏重时间近度

```python
session_search(query="用户 回复 对话 消息 2026-05-12", limit=3)
# 返回：3 条 cron session（同一天的调度器运行记录）
# 未返回：用户实际聊天记录
```

### 次轮：偏重用户行为

```python
session_search(query="最近 聊天 交互 2026 五月 5月", limit=3)
# 返回：用户 4/29 聊天 + 5/11 用户"想你了"聊天 + 5/5 cron
# ✅ 找到了被首轮遗漏的关键信息
```

### 生产验证数据（2026-05-12 10:00 BJT）

| 维度 | 首轮（时间偏重） | 次轮（行为偏重） |
|:----|:----------------|:----------------|
| 查询词 | "用户 回复 对话 消息 2026-05-12" | "最近 聊天 交互 2026 五月 5月" |
| 返回内容类型 | cron session 摘要 | 用户真实聊天 + cron session |
| 发现用户5/11聊天？ | ❌ 未发现 | ✅ 发现（"想你了"对话） |
| 时效性 | 同一天（5/12） | 跨天（4/29, 5/5, 5/11） |
| 交叉对比结论 | 不足：缺少用户实际交互 | ✅ 以时间最近的为准（5/11） |

**策略结论：** 当两次结果不一致时，以时间最近的结果为准——本 session 中次轮找到了 5/11 19:47 BJT 的用户聊天，晚于 state.json 记录的 null。

## 跨天 last_chat_time 校正模式

### 典型场景

1. 用户在 T 日通过飞书聊天（如说"想你了"）
2. 这个聊天创建了一个 feishu-source session
3. 状态文件未更新（因为没有任何调度器逻辑写入 last_chat_time）
4. 后续调度器运行一直以为 last_chat_time = null
5. session_search 在某次运行中偶然匹配到该 feishu session
6. **`last_chat_time` 从 null 更新为真实值**

### 为什么状态文件会遗漏

仅当以下情况时状态文件才更新 last_chat_time：
- 主动消息发送后更新（只更新 last_sent_time）
- 调度器手动写入（需 session_search 先发现用户聊天）

用户发起的 feishu 聊天**不会自动更新** active_contact_state.json。

### 关键发现：feishu-source session 的 session_id 格式

用户聊天的 session_id 格式为：
```
20260511_044717_c7758819
# 前缀：YYYYMMDD_HHMMSS_随机后缀
# 时区：PDT（UTC-7）
```

解释：session_id 中的时间戳 `044717` = 04:47:17 PDT = 11:47:17 UTC = **19:47 BJT**。因此用户实际聊天时间为 **2026-05-11 19:47 BJT**。

## 🧠 2026-05-12 11:01 BJT 验证：cron 上下文中 session_search 的行为特征

**背景：** 11:01 BJT 主调度器运行，查询词 "用户 回复 对话 消息 2026-05-12"，limit=3。

**结果：** 返回 3 条 cron session 摘要（09:00, 10:00, 10:30 的主调度器运行记录），**未返回任何用户实际聊天 session**。

### 为什么在 cron 上下文中 session_search 的结果偏「自指」？

| 原因 | 说明 |
|:----|:------|
| cron job 密度高 | 每30分钟运行一次，一天产生 48 条 cron session。这些 session 占据了搜索结果的「最新」位置 |
| 查询措辞偏系统 | 使用"用户 回复 对话 消息"虽然意在找用户交互，但 cron session 的摘要中通常也包含"用户""回复""对话"等高频词 |
| 用户交互稀疏 | 用户可能数天一次聊天，而 cron 每 30 分钟一次。从时间分布看，cron 条目远多于用户条目 |

### 对主调度器的影响

1. **session_search 在 cron 上下文中正常返回 cron session 是预期行为** — 不意味着搜索失败，只说明近期活动全是调度器运行
2. **当返回全是 cron session 时**：查看其中最新一条的摘要，确认用户是否有回复（若无回复，摘要通常会写明"用户未回复"）
3. **降级条件调整**：session_search 返回空（0 条）才是真正需要标记 [degraded-mode] 的情况。返回 cron session 不算降级
4. **交叉比对责任不变**：即使在 cron 上下文中，仍需将 session_search 发现的用户交互时间与 state.json 交叉比对

### 验证方法

```python
# cron 上下文中的典型 session_search 结果特征
results = session_search(query="用户 回复 对话 消息 2026-05-12", limit=3)
# 期望输出格式：
# [
#   {"source": "cron", "summary": "...用户今日无任何新回复或新发起聊天..."},
#   {"source": "cron", "summary": "...用户未回复..."},
#   {"source": "cron", "summary": "...确认用户无新交互..."}
# ]
# 以上均为预期行为，不属于搜索失败
```

**经验：** 在 cron 上下文中，返回的 session_search 结果大多为 cron session 是正常现象。真正的「异常」是 session_search 返回 0 条结果（系统负载/故障导致搜索完全失败）。区分「返回全 cron」和「返回空」两个场景，能避免不必要的降级标记。

## 经验总结

1. **至少两轮 query**：首轮偏时间，次轮偏行为。不要相信单次结果。
2. **跨天检查**：session_search 可能发现数天乃至数月前的用户聊天。定期搜索时不要限定太窄的时间窗口。
3. **交叉比对**：session_search 结果必须与 state.json 交叉比对。本 session 证明了 state.json 的 `last_chat_time = null` 可能是完全错误的。
4. **feishu-source session 识别**：用户发起的飞书聊天 session 的 source 为 `feishu`（非 `cron`）。在 session_search 结果中区分 source 类型有助于判断是否用户真正交互。
5. **session_id 时区解码**：格式 `YYYYMMDD_HHMMSS` 为 PDT (UTC-7)。需要 +15h 才能换算到 BJT。
